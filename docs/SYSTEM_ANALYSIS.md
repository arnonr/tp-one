## 2. ภูมิหลังและความจำเป็น

### 2.1 ภูมิหลัง

อุทยานเทคโนโลยี เป็นหน่วยงานรับบริการวิชาการของมหาวิทยาลัย อยู่ภายใต้การกำกับของรัฐ ให้บริการวิชาการครบ 4 ด้าน:

1. **เช่าพื้นที่/ห้องประชุม** — บริการเช่าพื้นที่สำนักงาน ห้องประชุม พื้นที่ coworking
2. **ที่ปรึกษา/วิจัย** — บริการที่ปรึกษา วิจัย พัฒนานวัตกรรม แก่องค์กรภายนอก
3. **อบรม/สัมนา** — จัดอบรม สัมนา เวิร์คช็อป ให้ความรู้แก่ภาคีเครือข่าย
4. **บ่มเพาะ/Incubation** — บ่มเพาะสตาร์ทอัป โปรแกรมเร่งรัดธุรกิจ (Accelerator)

### 2.2 ปัญหาปัจจุบัน

| ปัญหา              | รายละเอียด                                   | ผลกระทบ                           |
| ------------------ | -------------------------------------------- | --------------------------------- |
| ไม่มีระบบกลาง      | งานกระจายอยู่ใน LINE/Telegram, Excel, กระดาษ | ข้อมูลไม่เป็นศูนย์กลาง สูญหายง่าย |
| ติดตามงานไม่ได้    | ไม่รู้สถานะงาน งานตกหล่น                     | ประสิทธิภาพลดลง งานล่าช้า         |
| รายงานผู้บริหารยาก | ต้องรวบรวมข้อมูลทำรายงานด้วยมือ              | เสียเวลา ข้อมูลไม่ทันสมัย         |

### 2.3 เป้าหมายของระบบ

1. สร้างระบบกลางสำหรับจัดการงานทั้งหน่วยงาน
2. มอบหมาย ติดตาม และรายงานสถานะงานได้อย่างชัดเจน
3. จัดการโครงการและติดตาม KPI ของหน่วยงาน
4. สร้าง Dashboard และรายงานสำหรับผู้บริหาร
5. Self-host บน server มหาวิทยาลัย
6. รองรับ **ปีงบประมาณไทย (ต.ค. - ก.ย.)** ทั้งในระบบกรองข้อมูล รายงาน และ Dashboard

## 3. ขอบเขตของระบบ

### 3.1 ขอบเขตภายใน (In Scope)

- ระบบจัดการงาน (Task Management) สำหรับผู้บริหารและเจ้าหน้าที่
- ระบบจัดการโครงการ (Project Management)
- ระบบติดตาม KPI
- Dashboard สำหรับผู้บริหาร
- ระบบรายงาน (PDF/Excel Export)
- ระบบแจ้งเตือน (Telegram Bot API + Email)
- ระบบยืนยันตัวตนผ่าน SSO มหาวิทยาลัย
- แม่แบบงานมาตรฐาน (Task Templates)
- ระบบติดตาม "ของค้าง" (Waiting for Others)
- Quick Copy to Clipboard (สรุปงานส่ง Telegram)
- หน้า "งานของฉัน" (My Work Today)
- Quick Note Sidebar (จดบันทึกด่วน + แปลงเป็น Task)
- Snapshot of Success (ถ่ายรูป/แนบรูปหลักฐานกิจกรรม)
- Audit Timeline & Bottleneck Detection (Timeline งาน + ตรวจจับคอขวด)

### 3.2 ขอบเขตภายนอก (Out of Scope)

- ระบบจองห้องประชุม (อาจพัฒนาในเฟสถัดไป)
- ระบบการเงิน/การเช่าพื้นที่ (อาจพัฒนาในเฟสถัดไป)
- พอร์ทัลสำหรับผู้ใช้บริการภายนอก
- แอปพลิเคชันมือถือ (Mobile App)

## 4. กลุ่มผู้ใช้งาน

### 4.1 จำนวนผู้ใช้

ประมาณ 20-100 คน (เจ้าหน้าที่ภายในหน่วยงาน)

### 4.2 บทบาทและสิทธิ์ (RBAC)

ระบบใช้ **2 บทบาทระดับระบบ (Global Role)** + **สิทธิ์ระดับ Workspace/Project**

| บทบาทระดับระบบ | จำนวนประมาณ | หน้าที่หลัก                                          |
| -------------- | ----------- | ---------------------------------------------------- |
| **Admin**      | 1-3 คน      | จัดการระบบทั้งหมด จัดการผู้ใช้ ดูข้อมูลทุก workspace |
| **Staff**      | 15-90 คน    | ทำงานตามสิทธิ์ที่ได้รับในแต่ละ workspace/project     |

สิทธิ์เฉพาะเจาะจงกำหนดที่ระดับ **Workspace** และ **Project**:

**สิทธิ์ระดับ Workspace** (`workspace_members`):
| สิทธิ์ | คำอธิบาย |
|-------|---------|
| owner | สร้าง workspace เป็นเจ้าของ แก้ไข/ลบ workspace ได้, เพิ่ม/ลบสมาชิก, กำหนดสถานะ |
| editor | สร้าง/แก้ไขงานใน workspace, มอบหมายงาน, เปลี่ยนสถานะ |
| viewer | ดูข้อมูลใน workspace ได้อย่างเดียว |

**สิทธิ์ระดับ Project** (`project_members`):
| สิทธิ์ | คำอธิบาย |
|-------|---------|
| owner | แก้ไข/ลบโครงการ เพิ่ม/ลบสมาชิก จัดการ KPI |
| member | สร้าง/แก้ไขงานในโครงการ อัปเดตสถานะ |
| viewer | ดูข้อมูลโครงการได้อย่างเดียว |

### 4.3 ตารางสิทธิ์ (Global Role + Workspace/Project Permission)

### 5.6 ระบบติดตาม "ของค้าง" (Waiting for Others)

งานราชการมักติดขัดเพราะต้องรอหน่วยงานอื่นดำเนินการ (กองคลัง, พัสดุ, สำนักงานอธิการบดี ฯลฯ) เจ้าหน้าที่ต้องการ "ยัน" ได้ว่างานไม่ได้ค้างที่ตนเอง

**ปัญหา:** เวลาถูกตามงาน เจ้าหน้าที่ไม่มีหลักฐานชัดเจนว่า "รอใคร" และ "เอกสารอะไร" ทำให้ดูเหมือนงานค้างอยู่ที่ตัวเอง

| ID         | ข้อกำหนด                                                                                                      | ความสำคัญ |
| ---------- | ------------------------------------------------------------------------------------------------------------- | --------- |
| FR-WAIT-01 | เพิ่มสถานะพิเศษ **"รอหน่วยงานอื่น"** ในงาน (นอกเหนือจาก custom statuses)                                      | สูง       |
| FR-WAIT-02 | เมื่อตั้งสถานะ "รอหน่วยงานอื่น" ต้องระบุ: หน่วยงานที่รอ, เหตุผล, เลขที่เอกสารอ้างอิง (ถ้ามี), วันที่ส่งเรื่อง | สูง       |
| FR-WAIT-03 | งานที่อยู่ในสถานะ "รอ" เกิน X วัน แจ้งเตือนให้เจ้าของงานติดตาม (configurable)                                 | สูง       |

### 6.6 Database Migration Strategy

| ข้อกำหนด        | รายละเอียด                                                             |

| Seed data       | มี seed script สำหรับ initial data (roles, default workspace statuses) |

**Migration workflow:**

```
1. แก้ schema ใน packages/server/src/db/schema/*.ts
2. drizzle-kit generate → สร้าง SQL migration file
3. review SQL → commit migration file
4. drizzle-kit migrate (staging) → ทดสอบ
5. deploy production → drizzle-kit migrate (auto)
```

## 7. ปีงบประมาณไทย (Thai Fiscal Year)

### 7.1 หลักการ

ระบบราชการไทยใช้ **ปีงบประมาณ (Fiscal Year)** รอบตั้งแต่ **1 ตุลาคม ถึง 30 กันยายน** ไม่ใช่ปีปฏิทิน (ม.ค. - ธ.ค.)

ตัวอย่าง:

- **ปีงบประมาณ 2569** = 1 ต.ค. 2568 – 30 ก.ย. 2569
- **ปีงบประมาณ 2570** = 1 ต.ค. 2569 – 30 ก.ย. 2570

### 7.2 ผลกระทบต่อระบบ

### 7.4 Utility Function (ตัวอย่าง)

```
getFiscalYear(date):
  if date.month >= 10 (ต.ค.): return date.year + 1
  else: return date.year

getFiscalYearRange(fiscalYear):
  start: YYYY-10-01  (fiscalYear - 1)
  end:   YYYY-09-30  (fiscalYear)

getFiscalQuarter(date):
  month = date.month
  if month in [10,11,12]: Q1
  if month in [1,2,3]:    Q2
  if month in [4,5,6]:    Q3
  if month in [7,8,9]:    Q4
```

### 13.5.1 แหล่งข้อมูลเดิม

### 13.5.2 ขั้นตอน

1. **สำรวจข้อมูลเดิม** — เก็บรวบรวมไฟล์ Excel, รูปภาพ, เอกสารที่มีอยู่
